## Notes

These are not contract violations but are surfaced for the post-L2 synthesis pass.

1. **Notation hazard: `intra_cluster_edges[idx]` indexes by `oracle_idx`, not array position.** The `intra-cluster-edges.json` file contains 35 array entries; oracle_idx values run 4–6 and 40–76 (matching the L1 oracle's edge index). The catalog's notation `intra_cluster_edges[40]` therefore refers to the entry whose `oracle_idx` field equals 40, not to array index 40 (which doesn't exist — the array has only 35 entries, 0–34). This convention is consistent across the catalog and produces unambiguous lookups (every cited idx resolves to exactly one entry), but a reader unfamiliar with the convention could be confused. The synthesis pass may want to standardise on either `intra_cluster_edges[oracle_idx=40]` or `oracle.edges[40]` to make the convention explicit. Defect-equivalent in formality but not in substance.

2. **Discovery §4 footnote line ("does NOT directly import from `engine/`") is structurally a §7 deferral.** It appears in `01-cluster-discovery.md` rather than `04-cluster-report.md` §7. The bullet is explicitly framed as "deferred to synthesis" and the same observation is repeated in report §7, so the deferral channel discipline is honoured in spirit; but a strict reading of the contract clause "NO claim crosses the cluster boundary except via the 'Cross-cluster observations for synthesis' deferral channel in 04-cluster-report.md §7" would put this *only* in the report. The cluster's discipline is conservative (it tags the bullet with "Implicit cross-cluster facts (deferred to synthesis):"), so this is a stylistic note rather than a defect.

3. **Cluster report's bug-hiding-rule enumeration is incomplete.** Report §3 and discovery §5 both say "all on bug-hiding-pattern rules R1, R5, R6, R9." Empirically, the web/ violations also include R2 ×3, R7 ×4, R8 ×2 (all bug-hiding rules per `enforce_tier_model.py`'s ruleset, just less-frequently-cited ones). The substantive claim — *zero L1 layer-rule violations; all findings are bug-hiding patterns* — is correct. The enumeration of which specific bug-hiding rules fired is incomplete by three rules. Worth a one-line correction in the synthesis pass if the cluster report is revisited.

4. **Cluster total LOC arithmetic is internally consistent.** Discovery §2 footer states "Cluster total ~22,882 Python LOC" and report §1 states "Total in-scope: ~22,882 Python LOC across 75 files." Sum of per-subpackage LOC (1,072 + 1,224 + 1,952 + 407 + 8,189 + 3,748 + 257 + 1,011 + 4,080 + 824) = 22,764 LOC; the report's 22,558 LOC for `web/` is a different (likely all-of-web-not-counting-frontend-but-including-`skills/`) measurement. The 22,558 vs 22,764 vs 22,882 numbers are *close* (within 1%), but not perfectly cross-consistent — likely due to whether `skills/` (85 extra LOC for composer/skills) is included and rounding. Not a defect for L2; flagged for synthesis-pass arithmetic check.

5. **Catalog C8 (web/sessions/) inbound-from-cluster table is slightly under-cited.** C8's "Internal coupling" lists `web/blobs → web/sessions` weight 4 (idx 52) but does not enumerate `web/secrets → web/sessions` weight 1 (oracle_idx 70) or `web/sessions → web/blobs` weight 2 (oracle_idx 73, an outbound from sessions to blobs that bypasses C8's own outbound list). Both are present in `temp/intra-cluster-edges.json` and would appear in a strict bidirectional listing. Not a contract violation (C8's "Internal coupling" is not specified to be exhaustive), but synthesis pass may notice these omissions when stitching the L2 graph.
