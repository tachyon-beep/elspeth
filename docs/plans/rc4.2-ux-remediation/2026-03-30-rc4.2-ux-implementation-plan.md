# RC4.2 UX Feedback — Architecture Review & Implementation Plan

Date: 2026-03-30
Branch: RC4.2-UX
Source: Architecture review of `2026-03-30-rc4.2-ux-feedback-requirements.md` (this folder) against current implementation

---

## 1. Executive Summary

The RC4.2 UX requirements spec contains 17 items across 4 priorities. This plan reviews the current implementation, identifies gaps, proposes architecture decisions for the two major new subsystems (blob manager, secrets), and organises work into four implementation waves with explicit sequencing.

**Current state:** The web layer has 20+ REST endpoints, 13 composer tools (5 discovery + 8 mutation), WebSocket-based execution progress, versioned immutable composition state, and a three-provider auth system. The frontend has 18 components across chat, inspector, execution, sessions, and auth modules, backed by 3 Zustand stores.

**Key findings:**
- Two requirements (REQ-UX-01 sending fix, REQ-UX-02 scroll button) are small, isolated fixes
- Two requirements (REQ-API-01 blob manager, REQ-API-02 secrets) are major new subsystems
- One requirement (REQ-UX-03 fork) is a significant feature touching session model, frontend, and composition state
- The remaining 12 requirements are small-to-medium enhancements to existing infrastructure
- No requirement conflicts with the existing architecture; all can be added incrementally

## 2. Gap Analysis

### 2.1 REQ-UX-01: Fix "Sending" State (P0) — SMALL

**Current:** `sessionStore.sendMessage()` sets `local_status: "pending"` on optimistic insert and sets `isComposing: true` at the same time. The `POST /messages` call blocks until the LLM composition loop completes, so `local_status` stays "pending" for the entire duration. The user sees "Sending..." on their message bubble AND the "Assistant typing..." indicator simultaneously — contradictory signals.

**Gap:** The frontend already has two independent state signals:
- `local_status` on the message object → controls the "Sending..." label on the bubble
- `isComposing` on the store → controls the `ComposingIndicator`

The bug is that both activate simultaneously and `local_status` isn't cleared until the POST returns. This is a **frontend state machine problem**, not a transport problem.

**Implementation constraint:** The requirement describes a three-phase model:
1. Message submitted → bubble shows "sending" (pending)
2. Server returns 200 → bubble clears, composing indicator takes over
3. Assistant response arrives → indicator disappears, response rendered

The current `POST /messages` endpoint is synchronous — the 200 status and response body arrive together when the LLM composition loop completes. There is no point at which the server has acknowledged without also providing the response. This means phases 2 and 3 collapse into a single event: the POST completing.

**Consequence:** With the sync endpoint, the pending phase (`local_status: "pending"`) spans the entire round-trip including LLM composition time (potentially 10–30s). The requirement's phase 2 transition ("server returns 200 → clear pending") only fires when the response body is also ready, making the "composing" phase zero-length. The user would see "Sending..." for 10–30 seconds, then the response appearing instantly with no composing indicator — which is a different UX problem.

**Fix (deviates from requirement — see rationale):** In `sessionStore.sendMessage()`:
1. Add the user message to the local message list with `local_status: "pending"`.
2. Set `isComposing: true` in the same frame. **Do NOT show the "Sending..." label when `isComposing` is true** — the `MessageBubble` component suppresses the pending label when the composing indicator is active. The message appears as normal text (no "Sending..." suffix).
3. Scroll the chat viewport to the newest message on the send action.
4. When the POST succeeds: clear `local_status`, add the assistant response, update composition state, clear `isComposing`.
5. When the POST fails: set `local_status: "failed"`, clear `isComposing` (enables retry button).

**Deviation rationale:** The requirement's two-phase UX (visible "sending" → visible "composing") requires a server acknowledgement that arrives before the response body. The current sync endpoint cannot provide this. Rather than introduce a transport change (SSE/chunked) for a state machine bug, this fix eliminates the contradictory display by suppressing the pending label when composing is active. The user sees: message appears → composing indicator → response. The internal `local_status` tracking is preserved for failure recovery.

**If the requirement owner considers the visible "sending" phase important:** That requires converting `POST /messages` to a streaming response (SSE). The first event would be `{type: "ack"}` (clears pending), subsequent events deliver the response (clears composing). This is a worthwhile enhancement but a larger change — propose it as a separate follow-up.

**Scope:** Frontend-only. ~30 lines changed in sessionStore + MessageBubble display logic.

**Files affected:**
- `frontend/src/stores/sessionStore.ts` — `sendMessage()` state transitions, scroll on send
- `frontend/src/components/chat/MessageBubble.tsx` — suppress "Sending..." label when `isComposing` is true
- `frontend/src/components/chat/ChatPanel.tsx` — scroll to bottom on send action

**Testing prerequisite:** The repo currently lacks a frontend unit/component
test runner. Install and standardize `vitest` + `@testing-library/react`
before implementing this requirement so the send-state fix can ship with
real store/component coverage.

### 2.2 REQ-API-01: Blob Manager (P0) — MAJOR

**Current:** `POST /api/sessions/{id}/upload` accepts multipart upload, stores file at `{data_dir}/uploads/{user_id}/{filename}`, returns `{path, filename, size_bytes}`. No metadata table, no content hashing, no schema inference, no blob lifecycle, no output blob creation. The path is injected into chat as a string; the assistant uses it directly in `set_source(options={path: ...})`.

**Gap:** Complete subsystem. Need:

| Component | Status | Work |
|-----------|--------|------|
| `blobs` table in sessions.db | Missing | New SQLAlchemy table, 12 columns |
| Blob CRUD REST endpoints (6) | Missing | New router module |
| Blob composer tools (8) | Missing | New tool definitions + handlers |
| Schema inference service | Missing | CSV/JSON/JSONL column detection |
| Pipeline input wiring (`set_source_from_blob`) | Missing | Blob→source path resolution |
| Pipeline output wiring (`set_output_to_blob`) | Missing | Placeholder blob + post-run update |
| Blob manager drawer (frontend) | Missing | New component |
| Chat drag-and-drop integration | Partial | Upload exists; needs blob creation path |
| Upload endpoint migration | Exists | Deprecate in favour of blob creation |

**Architecture decisions:**

1. **Blob ownership: session-scoped.** Blobs belong to a session, not to a user globally. This aligns with the existing architecture: all data endpoints are under `/api/sessions/{id}/...`, ownership checks flow through session ownership, execution guardrails reference session-scoped paths, and session deletion cascades to blobs. The `blobs` table has a foreign key to `sessions(id)` with `ON DELETE CASCADE`.

   **Fork inheritance (REQ-UX-03 interaction):** When a session is forked, the fork endpoint copies blobs referenced by the inherited composition state into the new session. This is a **full copy** — new blob IDs, new storage paths, new files on disk. Simple and correct: each session fully owns its blobs, session deletion cascades cleanly with no cross-session dependencies, no reference counting, no garbage collector. The cost is disk duplication, which is acceptable for the file sizes involved (CSV/JSON uploads, typically < 100 MB). If this becomes a problem at scale, content-addressed deduplication can be added later without changing the ownership model.

2. **Blob storage location:** `{data_dir}/blobs/{session_id}/{blob_id}_{filename}` as specified. Distinct from `uploads/` — the upload directory becomes internal to blob creation, not user-facing. On fork, copied blobs get new storage paths in the new session's directory (`{data_dir}/blobs/{forked_session_id}/{new_blob_id}_{filename}`) and the files are duplicated on disk.

3. **Blob table placement:** In `sessions.db` (not Landscape). Blobs are user-facing lifecycle objects, not audit records. The Landscape records the *hash* of data that entered a pipeline, not the blob metadata.

4. **Source path allowlist update:** The S2/C3 path allowlist currently restricts sources to `{data_dir}/uploads/`. After blob manager, sources resolved via `set_source_from_blob` reference `{data_dir}/blobs/`. Do not patch this in three separate places. Extract a shared helper such as `_allowed_source_directories(settings)` and use it in composer tool validation, execution validation, and execution runtime guard together. Additionally, `set_source_from_blob` should validate that the `blob_id` exists and belongs to the current session (IDOR check at the tool level), then resolve the path internally. Direct `set_source` calls with raw paths still go through the allowlist.

5. **Output blob lifecycle:** `set_output_to_blob` creates a placeholder blob (`status: "pending"`) and wires the sink to write to its storage path. Post-execution, a hook updates the blob metadata (`status: "ready"`, `size_bytes`, `content_hash`). This requires the execution service to know about blobs — a new callback in the run completion path.

6. **Schema inference:** Separate service, not inline in blob creation. Called explicitly via `infer_blob_schema` tool or `POST .../infer-schema` endpoint. Reads first N rows (configurable, default 100), returns column names, inferred types, nullability, sample values. Supports CSV (via csv.Sniffer + pandas), JSON array, JSONL.

7. **Blob→PayloadStore relationship:** Blobs are user-facing; PayloadStore is audit infrastructure. When a pipeline reads from a blob, the engine records the raw data hash in PayloadStore as normal. The blob's `content_hash` and the PayloadStore hash should be produced by the same shared hash helper so they are guaranteed to match when the bytes match. Output blobs get their hash from the sink output, which also goes into PayloadStore.

8. **Blob→run linkage:** Use a normalized join table rather than JSON arrays on blob rows. This avoids non-atomic read-modify-write updates when multiple run-link events occur.

9. **Fork disk-growth guard:** Because blobs are copied on fork, add a session-level blob quota before forking ships so copy-on-fork has a balancing mechanism.

**Scope:** New `web/blobs/` subpackage (routes, service, protocol, schemas, models) + 8 new composer tools + frontend blob manager drawer + execution post-run hook.

**Files affected (new):**
- `src/elspeth/web/blobs/` — New subpackage (routes.py, service.py, protocol.py, schemas.py)
- `src/elspeth/web/sessions/models.py` — New `blobs` table
- `src/elspeth/web/composer/tools.py` — 8 new tool definitions
- `frontend/src/components/blobs/BlobManager.tsx` — New drawer component
- `frontend/src/stores/blobStore.ts` — New Zustand store (or extend sessionStore)

**Files affected (modified):**
- `src/elspeth/web/app.py` — Register blob router, pass blob service
- `src/elspeth/web/execution/service.py` — Post-run blob status update
- `frontend/src/components/chat/ChatInput.tsx` — Upload→blob creation path
- `frontend/src/components/chat/ChatPanel.tsx` — Drag-and-drop blob creation

### 2.3 REQ-API-02: Secret References (P1) — MAJOR

**Current:** No user-facing secret management. The only secret in WebSettings is `secret_key` (JWT signing). Auth credentials are handled by auth providers (bcrypt for local, OIDC tokens for SSO). Pipeline config that needs API keys (e.g., OpenRouter, Azure) has no resolution mechanism — the user would have to hardcode secrets in pipeline YAML, which is unacceptable.

**Gap:** Complete subsystem. Need:

| Component | Status | Work |
|-----------|--------|------|
| Secret resolution + storage | Missing | Protocol + two backends (DB for user, env for server) |
| `secrets` table or encrypted file | Missing | New persistence layer |
| Secret CRUD REST endpoints (4) | Missing | New router module |
| Secret composer tools (3) | Missing | New tool definitions |
| Runtime secret resolution service | Missing | Scoped lookup (user→org→server) |
| Secret reference format in config | Missing | `{"secret_ref": "NAME"}` detection/resolution |
| Frontend secrets management UI | Missing | Settings panel component |
| Validation integration | Missing | Check secret refs exist before execution |

**Architecture decisions:**

**Design-first: the resolution contract.** The secret system is three concerns that must be kept separate: (1) a resolution protocol that the execution layer calls, (2) storage backends behind that protocol, and (3) REST/tool APIs for managing user-scoped entries. Design the protocol first; storage is an implementation detail behind it.

1. **SecretResolver protocol:**
   ```python
   class SecretResolver(Protocol):
       def resolve(self, name: str, user_id: str) -> ResolvedSecret | None: ...
       def list_refs(self, user_id: str) -> list[SecretRef]: ...
       def validate_ref(self, name: str, user_id: str) -> bool: ...
   ```
   `ResolvedSecret` carries `name`, `value`, and `scope` (which scope satisfied the lookup). The `scope` is recorded in the audit trail; the `value` is consumed by the engine and never persisted or returned. Prefer designing this as an extension or superset of the existing `SecretLoader` / `CompositeSecretLoader` model in `core.security`, not as a parallel incompatible hierarchy. `ResolvedSecret` must also use a non-leaking `__repr__`/debug representation so plaintext values cannot escape through logs or tracebacks.

2. **Two-backend resolution (v1):**
   - **User-scoped secrets:** Stored encrypted in `sessions.db` (`user_secrets` table). Fernet encryption with key derived from server `secret_key` + per-user salt. User CRUD via REST API.
   - **Server-scoped secrets:** Read from environment variables with a configured prefix (e.g., `ELSPETH_SECRET_*`). Read-only — no CRUD via API. Listed in the API inventory with `scope: "server"`.
   - **Org-scoped secrets:** Deferred to v2 (when workspace/team infrastructure exists). The `SecretResolver` protocol supports it; no backend implements it yet.

   The resolver chains backends in priority order: `user → org → server`. First match wins. If no backend resolves the name, `resolve()` returns `None` and validation/execution fails with a clear error identifying the missing reference.

3. **Table schema:** The `user_secrets` table stores only user-scoped secrets. It does NOT attempt to represent server or org secrets — those come from their own backends. This avoids the inconsistency of requiring `user_id` and `encrypted_value` on rows that represent environment variables.

   ```sql
   CREATE TABLE user_secrets (
       id              TEXT PRIMARY KEY,
       name            TEXT NOT NULL,
       user_id         TEXT NOT NULL,
       encrypted_value BLOB NOT NULL,
       salt            BLOB NOT NULL,
       created_at      DATETIME NOT NULL,
       updated_at      DATETIME NOT NULL,
       UNIQUE(name, user_id)
   );
   ```

4. **Where resolution happens:** The `SecretResolver` is called inside `_run_pipeline()` (execution service background thread), after generating the settings dict from YAML and before passing to the Orchestrator. A `resolve_secret_refs(settings_dict, resolver, user_id)` function walks the config tree, finds `{"secret_ref": ...}` patterns, and replaces them with resolved values. This keeps plaintext values confined to the background thread — they never appear in the async route handler, composition state, or any response. Because this path runs in a worker thread, the user-secret backend used there must expose a synchronous access seam or a deliberately bridged thread-safe path.

   This is NOT a web-only code path: the `SecretResolver` protocol lives in `contracts/` (L0) and the `resolve_secret_refs()` tree-walk helper lives in `core/` (L1). The web layer provides the backends; the CLI can compose its own chain with an env-only backend.

5. **Integration with composition:** Composer tools see `{"secret_ref": "NAME"}` as opaque references. The `list_secret_refs` tool returns names and scopes (from all backends) but never values. Validation checks that referenced secrets exist and are accessible to the user via `resolver.validate_ref()`. Execution resolves them server-side.

6. **Audit/source blocker:** `SecretResolutionInput` currently only permits `source="keyvault"`. Extend allowed sources to include at least `env` and `user` before wiring runtime secret resolution into web execution. This is an implementation dependency, not a follow-up enhancement.

7. **Security invariant (unchanged):** The plaintext secret value NEVER appears in: composition state, chat messages, tool results, REST responses (except the write-only POST), WebSocket events, or browser-readable state. The frontend secrets UI is write-only — value field clears immediately after submission.

**Scope:** Protocol + resolution helper in `contracts/` and `core/`, web-specific backends + REST API in `web/secrets/`, frontend settings panel.

**Package layout — cross-surface reuse from day one.** The resolution protocol and tree-walk helper live outside `web/` so the CLI can reuse them without importing web infrastructure. The web layer provides the backends and the REST API.

**Files affected (new):**
- `src/elspeth/contracts/secrets.py` — `SecretResolver` protocol, `ResolvedSecret`, `SecretRef` (L0 — no upward imports)
- `src/elspeth/core/secrets.py` — `resolve_secret_refs(settings_dict, resolver, user_id)` tree-walk helper, `ChainedSecretResolver` (L1 — imports L0 only)
- `src/elspeth/web/secrets/` — Web-specific subpackage:
  - `user_store.py` — Fernet-encrypted DB backend implementing `SecretResolver`
  - `server_store.py` — Environment variable backend implementing `SecretResolver`
  - `routes.py` — REST endpoints (user secret CRUD + inventory listing)
  - `schemas.py` — Request/response Pydantic models
- `src/elspeth/web/sessions/models.py` — New `user_secrets` table
- `frontend/src/components/settings/SecretsPanel.tsx` — New component
- `frontend/src/stores/secretsStore.ts` — New Zustand store

**Files affected (modified):**
- `src/elspeth/web/app.py` — Construct resolver chain (user_store + server_store → ChainedSecretResolver), register secrets router
- `src/elspeth/web/composer/tools.py` — 3 new tools (`list_secret_refs`, `validate_secret_ref`, `wire_secret_ref`)
- `src/elspeth/web/execution/service.py` — Call `resolve_secret_refs()` (from `core.secrets`) in `_run_pipeline()`
- `src/elspeth/web/execution/validation.py` — Call `resolver.validate_ref()` for each `secret_ref` in config

### 2.4 REQ-UX-03: Chat Bubble Edit and Fork (P1) — SIGNIFICANT

**Current:** `MessageBubble.tsx` shows retry on failed messages only. No edit capability. Sessions are independent — no parent/child relationship, no fork concept. `SessionRecord` has no `forked_from` field.

**Gap:**

| Component | Status | Work |
|-----------|--------|------|
| Session fork endpoint | Missing | New endpoint with state/history duplication |
| `forked_from_session_id` + `forked_from_message_id` on sessions | Missing | Schema migration |
| Edit mode on MessageBubble | Missing | Textarea overlay, submit→fork flow |
| Fork provenance display | Missing | "Forked from..." banner in session header |
| Composition state inheritance | Missing | Copy state at fork point, not latest |

**Architecture decisions:**

1. **Fork endpoint shape:** `POST /api/sessions/{session_id}/fork` with body `{from_message_id, new_message_content}`. Creates a new session with:
   - All messages up to (but not including) the forked message
   - The new (edited) message as a user message
   - Composition state copied from the version that was current **before** the forked message was sent (not the latest version, and not the version produced by the assistant's response to that message)
   - `forked_from_session_id` and `forked_from_message_id` fields set

2. **State provenance — precise semantics.** Add `composition_state_id` to `chat_messages`. This field has different meaning depending on message role:

   - **User message:** `composition_state_id` = the state that existed **before** this message was sent. This is the "input context" — the pipeline the user was looking at when they typed this message. Recorded at the moment the user message is persisted (before the composition loop runs).
   - **Assistant message:** `composition_state_id` = the state that exists **after** the composition loop for this turn. This is the "output context" — the pipeline after the assistant's tool calls. Recorded when the assistant message is persisted (after composition completes and state is saved).

   **Fork reads the user message's `composition_state_id`** to find the state to copy. This is correct because forking from an edited user message means "go back to the pipeline state the user was looking at, then resubmit with different text." The assistant's modifications from that turn are discarded (they belong to the original session's timeline).

   **Ordering constraint:** the server should not infer this solely from "current state at save time." The client should send the state identity it was editing against, and the server should verify and persist that explicit pre-send context. This avoids races where concurrent changes move the session's current state between render time and send time.

   **Backfill:** Existing messages get `composition_state_id = NULL`. That is acceptable for pre-migration history only. For new post-migration messages, unexpected NULL provenance should be treated as a bug and test failure, not a graceful fallback.

3. **Blob inheritance (session-scoped, copy-on-fork).** Blobs are session-scoped (per REQ-API-01 architecture). When a session is forked:
   - The fork endpoint calls `BlobService.copy_blobs_for_fork(source_session_id, target_session_id, blob_ids)`.
   - This creates new blob records in the forked session with new IDs and new storage paths, and copies the underlying files to the new session's blob directory.
   - Each session fully owns its blobs. Session deletion cascades cleanly with no cross-session dependencies.

4. **Frontend flow:** Click edit on message → textarea replaces bubble text → submit → API fork call → new session created → sidebar updates → user switched to forked session → original session preserved in sidebar. A "Forked from {original_title}" banner appears in the forked session's header.

**Scope:** Backend session fork endpoint + schema migration + MessageBubble edit mode + session header provenance banner.

**Dependency:** Should be sequenced after REQ-API-01 (blob manager) to handle blob inheritance correctly. If blob manager isn't ready, fork can land without blob awareness — the `copy_blobs_for_fork` call becomes a no-op when no blobs exist.

### 2.7 REQ-API-03: Patch Operations (P2) — MEDIUM

**Current:** 8 mutation tools, all are set/upsert/remove operations. No partial update (patch) capability. To change one option on a source, the LLM must re-specify the entire `set_source` call.

**Gap:** 3 new composer tools + 3 new mutation methods on CompositionState.

**Architecture decisions:**

1. **Patch semantics:** JSON merge-patch per RFC 7396. `null` deletes a key, missing keys are untouched, non-null values overwrite. Shallow merge at the `options` level (not deep — matches the spec).

2. **Implementation on CompositionState:** New methods:
   - `with_source_options_patch(patch: dict) -> CompositionState` — merges patch into source.options
   - `with_node_options_patch(node_id: str, patch: dict) -> CompositionState` — merges patch into node.options
   - `with_output_options_patch(sink_name: str, patch: dict) -> CompositionState` — merges patch into output.options

3. **Validation:** Patch result must pass Stage 1 validation (same as all mutation tools — already enforced by ToolResult pattern).

**Files affected:**
- `src/elspeth/web/composer/state.py` — 3 new mutation methods
- `src/elspeth/web/composer/tools.py` — 3 new tool definitions + handlers

### 2.8 REQ-API-04: Atomic Full Pipeline Replacement (P2) — MEDIUM

**Current:** Building a pipeline from scratch requires N sequential tool calls (set_source + N * upsert_node + N * upsert_edge + N * set_output). Each triggers validation; intermediate states are invalid.

**Gap:** 1 new composer tool that atomically replaces entire composition state.

**Architecture decision:** `set_pipeline` takes `{source, nodes, edges, outputs, metadata}` and replaces the entire CompositionState in one operation. Internally, this constructs a new `CompositionState` from the provided data and validates it. The version increments once (not N times).

**Implementation:** New `CompositionState.from_pipeline_spec(source, nodes, edges, outputs, metadata)` class method + `set_pipeline` tool handler.

**Files affected:**
- `src/elspeth/web/composer/state.py` — New factory method

### 2.9 REQ-API-07: Enhanced Validation Model (P2) — MEDIUM

**Current:** `ValidationSummary` has `is_valid: bool` and `errors: tuple[str, ...]`. No warnings or suggestions. `ToolResult.to_dict()` returns `validation: {is_valid, errors}`.

**Gap:** Extend to three levels: errors (block execution), warnings (advisory), suggestions (optional improvements).

**Architecture decisions:**

1. **ValidationSummary extension:**
   ```python
   @dataclass(frozen=True, slots=True)
   class ValidationSummary:
       is_valid: bool
       errors: tuple[str, ...]
       warnings: tuple[str, ...]      # New
       suggestions: tuple[str, ...]   # New
   ```

2. **Warning examples:** "JSON sink path ends in .md", "output 'results' has no connected edge", "source has on_validation_failure='discard' — quarantined rows will be silently dropped".

3. **Suggestion examples:** "Consider adding an error sink for debugging", "Schema is not enforced — downstream field names may drift", "Gate has no route_false — non-matching rows will be dropped".

4. **Serialization:** `ToolResult.to_dict()` includes `validation: {is_valid, errors, warnings, suggestions}`. Empty arrays for warnings/suggestions when none apply.

5. **Persistence decision: transient only.** Warnings and suggestions are NOT persisted in `composition_states`. The DB schema continues to store only `is_valid` and `validation_errors` (the durable signals). Warnings and suggestions are computed fresh by `CompositionState.validate()` on each mutation and delivered to the assistant and frontend via the `ToolResult` response. Rationale: they go stale on the next mutation (a warning about a missing error sink disappears when one is added), so persisting them creates a stale-data problem. The frontend shows them from the most recent tool response, not from stored state.

6. **Scope boundary: Stage 1 only.** This requirement is about richer mutation responses (tool results) and inspector-facing status. The Stage 2 execution-time `ValidationResult` (which has `checks[]` and `errors[]`) does NOT need to change in this wave. Stage 2 validation runs real engine checks and its error model is structurally different (component-attributed errors with suggestions already). Extending it simultaneously would increase blast radius across backend schemas and frontend execution views without unlocking additional UX goals. Stage 2 can be extended later if warranted by its own requirement.

**Scope:** Stage 1 composition validation only. No DB schema changes. Should be done early in Wave 3 since other items (validation tint, tool response structure) depend on it.

**Files affected:**
- `src/elspeth/web/composer/state.py` — `ValidationSummary` extension (add `warnings`, `suggestions` tuples)
- `src/elspeth/web/composer/tools.py` — `ToolResult.to_dict()` serialization update
- `src/elspeth/web/sessions/routes.py` — `send_message` response includes validation from latest tool result (warnings/suggestions delivered alongside the assistant message)
- `frontend/src/types/index.ts` — Type updates for validation shape in tool responses
- `frontend/src/stores/sessionStore.ts` — Store latest warnings/suggestions from message response (transient state, not from composition_states)
- `frontend/src/components/inspector/SpecView.tsx` — Render warnings/suggestions from store's transient validation state (not from persisted composition state; `ValidationResult.tsx` is the Stage 2 execution component and is NOT affected)

### 2.17 REQ-API-06: Preview Execution (P3) — MEDIUM

**Current:** Execution runs full pipeline in background thread. No row-limiting or per-node output capture.

**Gap:** Two new tools: `preview_pipeline(max_rows)` and `preview_node_output(node_id, sample_rows)`.

**Architecture decisions:**

1. **Preview vs execution:** Preview must be explicitly non-destructive. It should NOT write to Landscape, NOT create a run record, and NOT write to sinks. It's a dry-run with data flow.
2. **Row limiting:** Source emits at most `max_rows` rows (configurable, default 5). Each node processes them normally.
3. **Per-node capture:** Intercept row state after each node. Return `{node_id: [row_dicts]}` for each node in the pipeline.
4. **Implementation approach:** Reuse the engine's Orchestrator with a `preview=True` flag that disables Landscape recording and sink writes, and enables per-node output capture. This is a significant engine change and should be carefully scoped.

**Risk:** Engine changes for preview mode could introduce complexity. Consider a simpler v1: validate + run first N rows through source only + return sample data. Full per-node preview in a later release.

**Files affected:**
- `src/elspeth/web/composer/tools.py` — 2 new tools
- `src/elspeth/web/execution/service.py` — Preview execution path
- Possibly `src/elspeth/engine/` — Preview mode support

## 3. Key Architecture Decisions Summary

| # | Decision | Rationale |
|---|----------|-----------|
| AD-1 | "Sending" fix suppresses pending label when composing is active; deviates from requirement's three-phase model | Sync POST can't deliver a server ack before the response body. Visible "sending" → "composing" transition requires SSE (separate follow-up). |
| AD-2 | Blobs scoped to session with copy-on-fork | Aligns with existing session-centric authz and cleanup. Fork copies both metadata and files. Simple ownership, no cross-session dependencies. |
| AD-3 | Blob storage at `{data_dir}/blobs/{session_id}/{blob_id}_{filename}` | Filesystem locality per session for cleanup; distinct from uploads/ |
| AD-4 | `SecretResolver` protocol in `contracts/` (L0), tree-walk helper in `core/` (L1), backends in `web/` (L3) | Layer-correct from day one. CLI can reuse protocol + resolver without web imports. Two distinct backends, one protocol. |
| AD-5 | User + server secret scopes for v1; org scope in protocol but no backend | Avoids building workspace/team infrastructure before it's needed |
| AD-6 | Secret resolution in `_run_pipeline()` background thread via `SecretResolver` | Never exposes plaintext in async context, route handlers, or responses. Resolution function is generic (not web-only). |
| AD-7 | `composition_state_id` on `chat_messages` with role-dependent semantics | User msg = state before send (fork input). Assistant msg = state after composition (fork excluded). Precise boundary enables correct fork. |
| AD-8 | JSON merge-patch (RFC 7396) for all patch operations | Standard, well-understood semantics; shallow merge at options level |
| AD-9 | ValidationSummary (Stage 1 only) extended with warnings + suggestions | Scoped to composition-time validation. Stage 2 execution validation unchanged — different error model, separate concern. |
| AD-10 | Shared drawer for blob manager + plugin catalog | Reduces layout complexity; natural grouping of reference panels |
| AD-11 | Preview execution deferred to minimal v1 (source preview only) | Full per-node preview requires engine changes; source preview is useful alone |

## 6. Risk Assessment

| Risk | Severity | Mitigation |
|------|----------|------------|
| **Blob manager scope creep** — schema inference, output wiring, and chat integration create a large surface area | High | Phase strictly: CRUD first, then tools, then output wiring, then frontend. Each phase is independently useful. |
| **Blob fork disk cost** — copy-on-fork duplicates files, which could be wasteful for large uploads | Low | Acceptable for expected file sizes (< 100 MB CSVs/JSONs). Content-addressed deduplication can be added later without changing the ownership model. |
| **Fork state-boundary semantics** — `composition_state_id` role-dependent meaning (before-send for user, after-compose for assistant) must be implemented precisely or forks will inherit wrong state | Medium | Enforce in `send_message` route handler: record state_id on user message BEFORE calling composer, record state_id on assistant message AFTER saving state. Unit test both paths explicitly. |
| **Secret resolution layer placement** — protocol and tree-walk helper must be usable outside the web surface | Low (mitigated) | Protocol in `contracts/` (L0), `ChainedSecretResolver` + `resolve_secret_refs()` in `core/` (L1). Web-specific backends in `web/secrets/`. CLI can compose its own chain (env-only backend) without importing web. |
| **Secret storage security** — encrypted secrets in SQLite are only as secure as the encryption key | Medium | Fernet with server secret_key is adequate for v1. Document that production deployments should use a strong secret_key and consider external KMS for v2. |
| **Preview execution engine coupling** — full per-node preview requires engine changes that may conflict with ongoing engine work | Medium | Defer to source-only preview for v1. Full preview is P3 and can be designed properly. |
| **Shared drawer complexity** — blob manager + catalog panel in one drawer may create UX confusion | Low | Separate tabs with clear labels. If it doesn't work, split into separate drawers — the backend is the same either way. |
| **Session DB migration** — adding columns/tables across blobs, secrets, and fork support requires migration infrastructure | Medium | Bootstrap or extend Alembic explicitly for `sessions.db` before Wave 1B/2A/2B schema work begins. |

## 7. New Database Schema (Cumulative)

Migration note: these schema changes assume a real migration path for
`sessions.db`, not ad hoc `create_all()` drift. Implementation should either
bootstrap a dedicated Alembic environment for the session database or extend an
existing migration framework explicitly before applying Wave 1B/2A/2B changes.

### New table: `blobs` (REQ-API-01)

```sql
CREATE TABLE blobs (
    id          TEXT PRIMARY KEY,                    -- UUID
    session_id  TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
    user_id     TEXT NOT NULL,                       -- Denormalized for query efficiency
    filename    TEXT NOT NULL,
    mime_type   TEXT NOT NULL,
    size_bytes  INTEGER NOT NULL,
    content_hash TEXT,                               -- SHA-256, NULL until computed
    storage_path TEXT NOT NULL,                      -- Internal, never exposed
    created_at  DATETIME NOT NULL,
    created_by  TEXT NOT NULL CHECK (created_by IN ('user', 'assistant', 'pipeline')),
    source_description TEXT,                         -- "uploaded", "pasted in chat", etc.
    schema_info JSON,                                -- Inferred columns/types
    status      TEXT NOT NULL DEFAULT 'ready' CHECK (status IN ('ready', 'pending', 'error'))
);
CREATE INDEX idx_blobs_session ON blobs(session_id);
CREATE INDEX idx_blobs_user ON blobs(user_id);
```

```sql
CREATE TABLE blob_run_links (
    blob_id      TEXT NOT NULL REFERENCES blobs(id) ON DELETE CASCADE,
    run_id       TEXT NOT NULL REFERENCES runs(id) ON DELETE CASCADE,
    direction    TEXT NOT NULL CHECK (direction IN ('input', 'output')),
    PRIMARY KEY (blob_id, run_id, direction)
);
CREATE INDEX idx_blob_run_links_run ON blob_run_links(run_id);
```

### New table: `user_secrets` (REQ-API-02)

User-scoped secrets only. Server-scoped secrets come from environment variables (no DB row). Org-scoped secrets deferred.

```sql
CREATE TABLE user_secrets (
    id              TEXT PRIMARY KEY,                -- UUID
    name            TEXT NOT NULL,
    user_id         TEXT NOT NULL,
    encrypted_value BLOB NOT NULL,                   -- Fernet-encrypted
    salt            BLOB NOT NULL,                   -- Per-secret salt
    created_at      DATETIME NOT NULL,
    updated_at      DATETIME NOT NULL,
    UNIQUE(name, user_id)
);
CREATE INDEX idx_user_secrets_user ON user_secrets(user_id);
```
