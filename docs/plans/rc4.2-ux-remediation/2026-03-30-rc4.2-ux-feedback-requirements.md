# RC4.2 UX Feedback — Requirements Spec

Date: 2026-03-30
Branch: RC4.2-UX
Source: User feedback on web composer MVP + assistant authoring API surface review

---

## Context

The web composer MVP landed in RC4.2. This spec captures the first round of user feedback and the results of reviewing the proposed assistant-facing API surface against what's implemented. It covers UX bugs, missing infrastructure (blob manager, secrets), frontend polish, and new composer tools.

---

## REQ-UX-01: Fix "Sending" state during assistant response

**Problem:** After sending a message, the user message bubble shows "sending" for the entire duration of the assistant's composition loop. The user sees "sending" even though the server acknowledged the message and the assistant is actively working.

**Current behaviour:** `sendMessage()` sets `local_status: "pending"` on the optimistic insert. Status stays pending until the full assistant response returns. The `ComposingIndicator` ("Assistant typing...") appears simultaneously, creating a contradictory display — the message appears unsent while the assistant is visibly responding.

**Required behaviour:** Two-phase status:

1. Message submitted → bubble shows "sending" (pending)
2. Server returns 200 → bubble clears to normal, `ComposingIndicator` takes over
3. Assistant response arrives → indicator disappears, response bubble rendered

Submitting a message must also scroll the chat viewport to the newest message immediately. This is distinct from the "user scrolled up reading history" case: an explicit send action means the user's focus has returned to the live bottom of the conversation.

The fix is in the send-flow UI state machine in `sessionStore.sendMessage()` and `MessageBubble.tsx`. The user message must not continue to present as "unsent" while the assistant is visibly composing a response.

**Implementation note (added during architecture review):** The three-phase model above (pending → composing → response) assumes the server returns 200 before the response body is ready. The current `POST /messages` endpoint is synchronous — 200 and the response body arrive together when the LLM composition loop completes. With this endpoint, phase 2 and phase 3 are the same event, making the "composing" phase zero-length. The implementation plan addresses this by suppressing the "Sending..." label on the bubble whenever the composing indicator is active, while keeping failure/retry semantics intact. This eliminates the contradictory display without a transport change. A visible pending → composing transition would require converting the endpoint to a streaming response (SSE), which is proposed as a separate follow-up enhancement.

**Priority:** P0

## REQ-UX-02: Scroll-to-bottom button

**Problem:** When the user scrolls up to read history, new messages arrive below the viewport with no way to jump back down.

**Current behaviour:** `ChatPanel.tsx` auto-scrolls to bottom unless user has scrolled up past a 40px threshold. No manual affordance to return.

**Required behaviour:** When the user is scrolled up past threshold, show a floating button at bottom-center of the chat panel. Clicking it snaps to the bottom and re-enables auto-scroll. The button disappears when already at bottom. Badge showing count of unseen messages since scrolling up is desirable but not required for v1.

**Priority:** P1

## REQ-UX-03: Chat bubble edit and fork from here

**Problem:** If the user makes a mistake or wants to try a different approach, they have to start a new conversation. There's no way to edit a previous message and branch from that point.

**Current behaviour:** `MessageBubble.tsx` shows a retry button on failed messages only. No edit capability on successful messages.

**Required behaviour:**

- Each user message bubble gets an edit icon (visible on hover or always visible — design TBD).
- Clicking edit replaces the bubble text with an editable textarea, pre-populated with the original message.
- Submitting the edit **forks** the conversation from that point — a new session is created inheriting all history and composition state up to the edited message. The original session and its history are preserved intact.
- The user is switched to the forked session automatically. The original session remains accessible in the sidebar.
- The assistant explains the fork naturally: "I branched from your earlier message and continued from there."

**Design principle: Fork, don't roll back.** Prior history remains auditable. Executed jobs are not destructively invalidated. The user's intent is preserved as branching rather than mutation. This aligns with ELSPETH's audit-first philosophy — you never destroy evidence of what happened.

**Required provenance behaviour:**

- The original session remains unchanged and fully viewable.
- The forked session contains the inherited conversation up to the chosen message, then the edited replacement message and subsequent assistant output.
- The forked session must visibly indicate that it was created from an earlier session/message, for example via a "forked from..." banner, badge, or metadata row in the session header.
- Any later messages that existed after the fork point remain only in the original session; they are not copied into the fork.
- The forked session inherits composition state from the selected point in history, not from the latest state of the original session.

**Backend/API note:** A dedicated fork endpoint is likely, but the exact route shape is implementation detail. The product requirement is "fork from message with inherited state and preserved provenance," not a specific REST signature.

**Priority:** P1

## REQ-UX-04: Chat bubble copy icons

**Problem:** Users can't easily copy assistant responses.

**Required behaviour:** Each message bubble gets a copy-to-clipboard icon (top-right, visible on hover). Copies the text content only — tool call disclosures are excluded. Use the same "Copied!" toast pattern from `YamlView.tsx`.

**Priority:** P1

## REQ-UX-05: Version control restructure

**Problem:** The version dropdown in the inspector header is too wide and sits alongside the tabs at the same level. The relationship between versions and their content (spec, graph, yaml) isn't clear. Runs don't belong under a version but currently share the same tab level.

**Current behaviour:** `InspectorPanel.tsx` has a flat tab strip: `[Spec] [Graph] [YAML] [Runs]` with a version dropdown alongside.

**Required behaviour:**

```
┌─────────────────────────────────────┐
│  v3 ▾              [Validate] [Run] │  ← compact version selector + actions
│  ┌──────┬───────┬──────┬──────┐     │
│  │ Spec │ Graph │ YAML │ Runs │     │  ← tabs
│  └──────┴───────┴──────┴──────┘     │
│                                     │
│  [tab content]                      │
│                                     │
└─────────────────────────────────────┘
```

- Version selector is a compact control (shows "v3" not a full-width select box). Dropdown on click shows version history with timestamps and node counts.
- Spec, Graph, YAML show content for the selected version.
- Runs tab stays at the tab level but displays which version each run executed against (already does this).
- Revert action moves into the version dropdown (select an old version → "Revert to this version" button in dropdown).

**Priority:** P2

## REQ-UX-06: Validation status tint

**Problem:** There's no ambient indicator of whether the current pipeline version has been validated. The user has to click Validate and read the banner.

**Required behaviour:** The inspector panel (or the version selector, or the tab strip area) shows a subtle colour indicator:

- **Green:** Validation passed for this version
- **Red:** Validation failed for this version
- **Amber/yellow:** Not yet validated (new version, never validated)

This colour follows `validationResult` in the execution store plus a `never_validated` flag that's set when the composition version changes and cleared on first validation.

Implementation requirement: place the status indicator in one consistent location that is visible regardless of whether the user is reading Spec, Graph, or YAML. The preferred treatment is a thin stripe or dot attached to the compact version control area, not a full-panel background tint.

**Priority:** P2

## REQ-UX-07: Graph readability improvements

**Problem:** User feedback: "the graph manager is bad." Clarified intent: the graph is too small to read.

**Current behaviour:** `GraphView.tsx` uses React Flow with dagre auto-layout. Read-only. Nodes are small, edges are unlabelled, no coordination with the spec view.

**Required behaviour (minimum):**

1. Default graph presentation must be legible without manual zooming on a typical desktop viewport.
2. Nodes must be visibly larger, with readable text sizing for node name, plugin name, and type.
3. Initial fit/zoom behaviour must prioritise readability over showing the entire graph at any cost.
4. A one-click "fit graph" or "reset zoom" affordance must be available.

**Desirable follow-ons:**

1. Edge labels showing connection type, such as `on_success`, `route: true`, and `on_error`
2. Minimap for orientation in large graphs
3. Bidirectional highlight sync with SpecView

**Deferred:** Interactive graph editing (drag-to-connect, add/delete nodes). The assistant-first model means the graph is a view, not an editor.

**Priority:** P2

## REQ-API-01: Blob Manager

**Problem:** Users need to submit their own files and manage them via a file manager. Those files need to be usable as pipeline inputs and outputs. Currently, file upload drops a file to disk and injects a path string into chat — there's no file management, no structured reference model, and no way for pipeline outputs to appear as downloadable files.

### Resource model

A **Blob** is a user-managed file with metadata, scoped to a session and linked to the jobs that consume or produce it.

```
Blob {
  blob_id:             string (uuid)
  session_id:          string
  filename:            string (user-visible name)
  mime_type:           string (detected or declared)
  size_bytes:          int
  content_hash:        string (SHA-256)
  storage_path:        string (internal — never exposed to users or assistant)
  created_at:          datetime
  created_by:          "user" | "assistant" | "pipeline"
  source_description:  string | null   # "uploaded", "pasted in chat", "output from run {id}"
  schema_info:         JSON | null      # inferred columns/types for structured data
  status:              "ready" | "pending" | "error"
  run_links:           { run_id: string, direction: "input" | "output" }[]
}
```

Blobs are distinct from PayloadStore. PayloadStore is content-addressed audit infrastructure (hash-keyed, immutable). Blobs are user-facing objects with stable IDs, mutable metadata, and lifecycle. A pipeline run that reads from a blob still records the raw data hash in PayloadStore for the audit trail.

The key product requirement is seamless mapping between:

- chat and pasted/uploaded user inputs
- pipeline/job inputs
- pipeline/job outputs
- downloadable files in the blob manager

The user should never have to reason about storage paths to move between those states.

### Storage

```
data/blobs/{session_id}/{blob_id}_{filename}
```

### Database (sessions.db)

New `blobs` table with the fields above. Foreign key to `sessions(id)`.

### REST endpoints

| Method | Endpoint | Purpose |
|--------|----------|---------|
| `POST` | `/api/sessions/{id}/blobs` | Create blob — multipart upload OR JSON body with inline content |
| `GET` | `/api/sessions/{id}/blobs` | List blobs (metadata only) |
| `GET` | `/api/sessions/{id}/blobs/{blob_id}` | Blob metadata |
| `GET` | `/api/sessions/{id}/blobs/{blob_id}/content` | Download blob content |
| `DELETE` | `/api/sessions/{id}/blobs/{blob_id}` | Delete blob and file |
| `POST` | `/api/sessions/{id}/blobs/{blob_id}/infer-schema` | Detect columns/types |

**Create blob has two modes:**

1. **Multipart upload** — user drags a file or clicks upload. Content-Type: multipart/form-data.
2. **Inline content** — assistant materialises conversation content. Content-Type: application/json with `{filename, content_type, content}`.

Both return the same `BlobMetadata` response.

### Composer tools

| Tool | Type | Purpose |
|------|------|---------|
| `create_blob(filename, content, content_type)` | Mutation | Materialise conversation content into a managed blob |
| `list_blobs()` | Discovery | List session blobs with metadata |
| `get_blob_metadata(blob_id)` | Discovery | Metadata + inferred schema |
| `delete_blob(blob_id)` | Mutation | Remove a blob |
| `read_blob_text(blob_id, max_bytes?)` | Discovery | Read text content of a blob — useful for assistant confirmation and debugging |
| `infer_blob_schema(blob_id)` | Discovery | Detect structure for CSV/JSON/JSONL |
| `set_source_from_blob(blob_id, plugin?, on_success, on_validation_failure?)` | Mutation | Wire blob as pipeline source — resolves path, infers plugin from mime_type |
| `set_output_to_blob(sink_name, plugin?, filename?)` | Mutation | Configure sink to write output as a new blob |

### How blobs connect to pipelines and jobs

**Input flow:**
1. User uploads `customers.csv` → blob `b_123` created at known path
2. Assistant calls `set_source_from_blob(blob_id="b_123", on_success="raw_rows")`
3. Internally translates to `set_source(plugin="csv", options={path: <resolved>}, on_success="raw_rows")`
4. Source options carry a `blob_ref` field for traceability
5. The blob manager shows that this blob is being used as an input to the resulting job/run

**Output flow:**
1. Assistant calls `set_output_to_blob(sink_name="results", filename="rules.json")`
2. Creates a placeholder blob (`status: "pending"`) and wires sink to write to its path
3. Pipeline runs and writes to that path
4. Post-execution: blob metadata updated (`status: "ready"`, `size_bytes`, `content_hash`)
5. Blob appears in file manager as a downloadable result
6. The blob manager shows which run produced it

**Paste flow:**
1. User pastes JSON data into chat
2. Assistant calls `create_blob(filename="input.json", content="[...]", content_type="application/json")`
3. Then calls `set_source_from_blob(blob_id=<new_id>)`
4. User never sees a file path

### Schema inference

For structured blobs (CSV, JSON array, JSONL), `infer_blob_schema` returns:

```json
{
  "blob_id": "b_123",
  "detected_format": "csv",
  "columns": [
    {"name": "url", "inferred_type": "str", "nullable": false, "sample_values": ["https://..."]},
    {"name": "score", "inferred_type": "int", "nullable": true, "sample_values": [10, null]}
  ],
  "row_count_sampled": 100,
  "row_count_total": 5000
}
```

The assistant uses this to auto-populate `schema_config.fields` in the source configuration.

### Frontend

**Blob manager drawer:** Opens from a file-manager icon next to the chat input upload button. Contains:

- File list (grid or list) showing filename, type icon, size, age, source badge ("uploaded" / "assistant" / "pipeline output")
- Per-blob actions: download, delete, "use as input" (tells assistant to wire it)
- Drag-and-drop zone for uploads
- Pipeline output blobs appear automatically after runs complete, badged as outputs
- Status indicators: ready (normal), pending (pipeline hasn't written yet), error (run failed)
- Clear linkage to runs/jobs: "used by run ..." and "produced by run ..."

**Chat integration:**
- Drag file onto chat → creates blob + inserts reference message
- Upload button creates blob (replaces current raw upload)
- Assistant references blobs by filename in conversation
- User can explicitly pick an existing blob as the next input without re-uploading it

### Replaces

The current `POST /api/sessions/{id}/upload` endpoint is deprecated and subsumed by blob creation.

**Priority:** P0

## REQ-API-02: Secret reference handling

**Problem:** Users can't configure pipelines that need API keys (e.g. OpenRouter, Azure) because there's no way to provide secrets safely. No user should paste credentials into chat.

### Model

Any plugin config field that may contain a secret accepts either a literal value or a reference:

```json
{"secret_ref": "OPENROUTER_API_KEY"}
```

References are resolved server-side at validation/execution time only. The composer tools and web interface see and return the `secret_ref` structure and capability metadata, never the actual value.

### Three-tier secret scoping

1. **User-level secrets:** Set by user via dedicated UI. Stored encrypted at rest, scoped to user (not session — secrets persist across sessions for the same user).
2. **Org-level secrets:** Set by organisation admin. Available to members of the workspace.
3. **Server-level secrets:** Set by admin via server-side configuration. Available to all users.

### REST endpoints

| Method | Endpoint | Purpose |
|--------|----------|---------|
| `GET` | `/api/secrets` | List available secret names — all scopes. No values returned. |
| `POST` | `/api/secrets` | Create user-scoped secret (name + value). Value is write-only and encrypted at rest. |
| `DELETE` | `/api/secrets/{name}` | Delete user-scoped secret. Server/org secrets can't be deleted via API. |
| `POST` | `/api/secrets/{name}/validate` | Check if secret exists and is non-empty. No value returned. |

Response for list includes scope metadata:

```json
[
  {"name": "OPENROUTER_API_KEY", "scope": "user", "available_to_runtime": true},
  {"name": "AZURE_TENANT_ID", "scope": "server", "available_to_runtime": true}
]
```

### Composer tools

| Tool | Type | Purpose |
|------|------|---------|
| `list_secret_refs()` | Discovery | Names + scopes of available secrets |
| `validate_secret_ref(name)` | Discovery | Check existence and accessibility |
| `wire_secret_ref(name, scope?)` | Mutation | Place an opaque `{ "secret_ref": "NAME" }` marker into pipeline config |

Secret creation/update is a dedicated REST/UI operation, not a composer/chat tool. In the web interface, a user enters a secret value into a dedicated form; the value is submitted immediately to profile/org/server-backed secret storage and then disappears from readable UI state. After storage, only the reference name, scope, and availability remain visible. The assistant-facing composer tool is `wire_secret_ref`, which only places the opaque reference marker into pipeline config.

### Frontend

Settings modal or panel accessible from user menu. Simple form: name field, masked value field, save/delete. Shows server-level and org-level secrets as available references with read-only badges, and user-level secrets as manageable references. Secret entry UI is write-only — clears the value field immediately after successful submission. After submission, the UI shows that the named reference exists and is usable, but never the value.

### Security rules (non-negotiable)

1. No tool returns the plaintext secret to the assistant.
2. No web API returns the plaintext secret to the browser after submission.
3. Secret entry UI is write-only and clears immediately after successful submission.
4. Secret inventory UI shows that a named reference is available, never its value.
5. Validation errors identify missing or unusable references without leaking values.
6. Secret-backed mutations are auditable as references and scopes, not raw material.
7. The assistant may enumerate usable secret references but must never see contents.
8. Secret resolution happens server-side at use time; the browser never receives resolved material.

### Runtime secret resolution

ELSPETH needs a scoped secret-resolution service rather than a flat secret namespace.

Minimum supported scopes:

1. user/profile
2. org/workspace
3. server

Required behaviour:

1. List available secret references and their scopes to the assistant/web UX
2. Resolve the secret value only inside trusted server/runtime code
3. Apply policy-driven lookup when a scope is not explicit, for example `user -> org -> server`
4. Record which scope satisfied the lookup for audit/debug visibility without exposing the value
5. Block validation/execution if no valid secret is available for a required reference

The secret value must never enter composition state, chat messages, or any browser-readable response.

**Priority:** P1

## REQ-API-03: Patch operations for composer tools

**Problem:** The LLM frequently needs to change one field in a source/node/sink config (e.g. swap model ID, change output path) but currently must re-specify the entire configuration via `set_source`, `upsert_node`, or `set_output`. This wastes tokens and increases error rate — the LLM has to reproduce fields it isn't changing.

### New composer tools

| Tool | Type | Purpose |
|------|------|---------|
| `patch_source_options(patch)` | Mutation | JSON merge-patch on source options |
| `patch_node_options(node_id, patch)` | Mutation | JSON merge-patch on node options |
| `patch_output_options(sink_name, patch)` | Mutation | JSON merge-patch on output options |

**Semantics:** JSON merge-patch per RFC 7396. Shallow merge at the options level. Explicit `null` deletes a key. Non-null values overwrite. Missing keys are untouched.

**Example:** Change the model on an LLM transform node:
```json
patch_node_options("classify", {"model": "openrouter/anthropic/claude-sonnet-4"})
```
Instead of re-specifying the entire node with all its other options.

**Priority:** P2

## REQ-API-04: Atomic full pipeline replacement

**Problem:** When the assistant builds a complete pipeline from scratch (common in first-message flows), it currently needs to call `set_source` + N × `upsert_node` + N × `upsert_edge` + N × `set_output` sequentially. Each call triggers validation, and intermediate states are invalid (e.g. edges referencing nodes that haven't been added yet).

### New composer tool

| Tool | Type | Purpose |
|------|------|---------|
| `set_pipeline(source, nodes, edges, outputs, metadata)` | Mutation | Atomically replace the entire composition state |

Returns the new state with validation result. Single composition budget charge.

**Priority:** P2

## REQ-API-07: Enhanced validation model

**Problem:** Validation currently returns `is_valid` + `errors[]`. No warnings or suggestions. The assistant and user miss useful guidance.

**Required:** Extend `ValidationResult` to three levels:

- **errors[]** — blocks execution
- **warnings[]** — doesn't block, should be addressed (e.g. "JSON sink path ends in .md", "output 'results' has no connected edge")
- **suggestions[]** — optional improvements (e.g. "consider adding an error sink", "schema is observed — downstream field names may drift")

Every composer mutation tool should include `validation: {is_valid, errors, warnings, suggestions}` in its response. Stage 1 validation already runs after each tool call — this formalises the response structure.

**Priority:** P2

## REQ-UX-08: Plugin catalog panel

**Problem:** Users don't know what building blocks are available. They can't ask the assistant to "use a CSV source" if they don't know CSV sources exist. The assistant knows the catalog (via discovery tools), but the user is flying blind.

**Required behaviour:** A collapsible panel in the bottom-right of the screen with three tabs:

- **Sources** — lists every available source plugin
- **Transforms** — lists every available transform plugin (including gates, aggregations, coalesce)
- **Sinks** — lists every available sink plugin

Each entry shows:

- **Plugin name** (e.g. "CSV", "JSON", "LLM", "Azure Blob")
- **One-line summary** of what it does
- **Expandable detail** with key configuration options, required fields, and an example use case in plain language (e.g. "Reads rows from a CSV file. Supports custom delimiters, encoding, and schema validation.")

**Interaction:**

- Clicking a plugin entry could insert a suggestion into the chat input (e.g. "Use the CSV source to read from my uploaded file") or simply act as reference — design TBD.
- The panel should be collapsible/dismissable so it doesn't consume space once the user knows what they want.
- Panel defaults to collapsed on small viewports.

**Data source:** The catalog API already exists (`GET /api/catalog/sources`, `/transforms`, `/sinks` and `GET /api/catalog/{type}/{name}/schema`). The frontend needs to fetch and render it. The one-line summaries come from `PluginSummary` (already returned by the catalog endpoints). The expanded detail comes from `get_plugin_schema` — the description fields and required/optional breakdown.

**Layout:** Bottom-right overlay or docked panel, not a replacement for the inspector. Think of it as a reference drawer — always accessible, never in the way. Could share the same drawer mechanism as the blob manager (REQ-API-01) with a tab for "Catalog" alongside "Files".

**Priority:** P2

## Disregarded proposals

These were in the API surface review but are not being taken forward.

| Proposal | Reason |
|----------|--------|
| Transaction support (`begin/commit/rollback`) | Session/message forking and immutable version history cover the user-facing need more directly than transactional chat authoring. |
| `apply_pipeline_patch` (structured multi-patch) | `set_pipeline` covers atomic full-replace. Individual patch tools cover partial edits. Multi-patch adds complex merge semantics for minimal gain. |
| `clone_node(id, new_id)` | The LLM can read a node and `upsert_node` with a new ID. Not worth API surface. |
| `auto_wire_connections()` | Stage 1 validation already detects disconnected components. Auto-wiring could create incorrect graphs that look valid. |
| `find_unconnected_components()` | Covered by validation errors. |
| `suggest_pipeline_from_intent()` | This is what the composer does. Duplicates the entire composition loop. |
| `suggest_prompt_template()` | The LLM writes prompts directly in conversation. |
| `suggest_field_mapping()` | Same — handled in conversation. |
| `resolve_secret_capabilities(name)` | `list_secret_refs` metadata + `validate_secret_ref` covers the need. |
| `fork_session_from_version(version_id)` | Fork-from-message is the primary user-facing flow. Version-level fork may still exist as an internal/API primitive, but it is not needed as a first-class product requirement here. |

## Design conventions adopted

1. **Secret references:** `{"secret_ref": "NAME"}` in any config field that may contain a secret. Resolved at execution time only.
2. **Blob references:** Blobs are referenced by `blob_id` in composer tools. Internal path resolution is invisible to the assistant and user.
3. **Validation in every mutation:** Every composer tool that mutates state returns `validation: {is_valid, errors, warnings, suggestions}`.
4. **Patch semantics:** JSON merge-patch (RFC 7396) for all patch operations.
5. **Fork, don't roll back:** Conversational revision creates new branches. Prior history and executed work are never destructively rewritten.
