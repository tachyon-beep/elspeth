# Round 2 — Dissent: Sable (Security Architect)

## Target: Seren (Systems Thinker) — "Feedback Instrument, Not Gatekeeper"

## Steelman

Seren's analysis is the most sophisticated in the roundtable. The four system archetypes (Shifting the Burden, Eroding Goals, Fixes that Fail, Success to the Successful) are real dynamics that have killed security tools at organisations far more disciplined than ours. The core insight — that the tool's aggregate health signals (suppression rate trends, repeat offender patterns, violation velocity) are higher-leverage than individual pass/fail verdicts — is correct at the systems level. Individual findings are low-leverage (Meadows level 12, parameters); information flow changes (making suppression rates visible and alarming) are high-leverage (Meadows level 6). If forced to choose between a tool that blocks violations one at a time and a tool that shows you your trust boundary health is degrading, the second tool changes more behaviour.

The "Fixes that Fail" archetype is particularly well-applied: a tool that makes developers *feel* their trust boundaries are enforced while missing semantic adequacy is genuinely worse than no tool — it creates false confidence that displaces the human scrutiny that would otherwise operate. The conclusion — design the tool to surface system-level signals, not just individual gates — follows logically from the analysis.

This is the strongest version of Seren's position. And I believe it leads to the wrong design priority.

## Attack

Seren's framing contains a subtle but critical contradiction, and then draws the wrong conclusion from it.

**The contradiction.** Seren argues that the precision threshold must be "immutable in code, not configurable" because "making it configurable is an invitation to erode it." This is a gatekeeper argument — it says the tool should enforce a hard constraint that cannot be weakened by the people it governs. But Seren *also* argues the tool should be a "feedback instrument, not a gatekeeper" — its "primary output should be system-level health signals, not individual pass/fail verdicts." These two positions are structurally incompatible. An immutable precision threshold is a gate. A primary output of trend lines is an advisory. You cannot simultaneously argue for an unyielding gate *on the tool's own calibration* while arguing the tool's enforcement output should be primarily informational.

Seren might respond: "I don't oppose blocking — I want *both* blocking and trend reporting." Fair enough. But the framing matters. "Primarily a feedback instrument" is not a neutral architectural statement — it's a design priority that determines what gets built first, what gets cut when scope is constrained, and what the team thinks the tool *is*. If the tool is primarily a feedback instrument, the blocking gate is the afterthought. And afterthought gates get disabled first when they're inconvenient.

**The evidence from this codebase.** Seren's own archetype analysis predicts what happens to advisory tools. The Eroding Goals archetype says: goals (like the 95% precision threshold) erode under pressure. Now apply this to the tool's framing. If the tool is a "feedback instrument" that also happens to block, the first thing that erodes under pressure is the blocking behaviour. "The trends show we're improving, so let's make the gate advisory for this sprint." Trend reporting is a complementary output; it must not be the tool's *identity*.

The existing ELSPETH enforcer (`enforce_tier_model.py`) provides a concrete data point. It runs as a blocking CI gate. It catches 1–2 violations per day. Those violations get fixed — not because developers read trend reports, but because *the build is red until they do*. The discussion paper (§8.4) reports this as evidence the approach works. The working mechanism is enforcement, not feedback. Seren's proposal to reframe the tool as a feedback instrument would displace the exact mechanism that the existing tool demonstrates is effective.

**The security architecture objection.** From my perspective as a security architect, trust boundaries are not advisory. They are either enforced or they are not. A trust boundary that is "recommended but not enforced" is not a trust boundary — it is a suggestion. The entire ELSPETH data manifesto (Tier 1 crash on corruption, Tier 3 validate at boundary) is written as imperatives, not advisories. "Bad data in the audit trail = crash immediately" is a gate. The tool that enforces this manifesto must share its character.

Seren's trend reporting is valuable — but it is a *diagnostic* layer on top of enforcement, not a replacement for it. The suppression rate metric, the repeat offender patterns, the violation velocity tracking — all of these are excellent operational signals. But they derive their meaning from the gate. A suppression rate is only meaningful if there's something to suppress against. A violation velocity matters only if violations have consequences. Remove the gate and the metrics become academic.

**The deeper issue: who is the tool for?** Seren's framing optimises for the team lead or engineering manager — someone who looks at dashboards and makes structural decisions. That's a valid audience. But the tool's *primary* audience is the CI pipeline, making a binary decision on every commit: does this code respect trust boundaries, yes or no? The team lead's dashboard is a secondary output. Getting this priority wrong means the tool gets evaluated on its dashboard quality rather than its enforcement reliability — and that is how security tools die. They become "nice to have" rather than "cannot ship without."

**Addressing the scribe's observation on my heuristic list.** The scribe noted, via Pyre, that my undeclared external call detection depends on a heuristic list that is itself an attack surface. This is correct. An agent using an unlisted external library (say, `aiohttp` instead of `httpx`) would bypass detection entirely. My response: the heuristic list is a *default*, not a perimeter. The `strict.toml` manifest must support project-specific additions, and — critically — the tool should ship with a `strict manifest audit` command that scans import statements for modules that *look* like HTTP/IO/database libraries (matching patterns like `*.client`, `*.api`, `*.request`) and suggests additions. The list is never complete, but incompleteness is a known residual risk documented in the manifest, not a silent gap. The default-deny posture ("undeclared external calls are flagged") means the heuristic list determines which calls are *automatically* recognised, not which calls are *the only ones checked*. Any call to any function can still be manually declared as an external boundary in the manifest.

## Proposed Verdict

The roundtable should decide:

1. **The tool is a gatekeeper that also provides feedback.** Not the reverse. The blocking CI gate is the primary function. Trend reporting, suppression rate dashboards, and violation velocity tracking are secondary outputs built on top of the enforcement data. The framing in all design documents, user-facing help text, and documentation should reflect this priority.

2. **Seren's specific metric proposals should be adopted — as enforcement diagnostics.** The suppression rate as a first-class alarming metric is excellent and should ship in v0.1. The repeat offender pattern detection is valuable for v0.2. But these are *diagnostic signals about enforcement health*, not the tool's primary output. The distinction is: "your gate is catching a lot and that tells you something about your codebase" (diagnostic) vs. "here are some trends, act on them if you want" (advisory). The first framing preserves the gate; the second erodes it.

3. **The precision threshold should be immutable in the tool's default profile** (adopting Seren's own recommendation, which is consistent with gatekeeper framing and inconsistent with feedback-instrument framing). Projects can define custom profiles with lower thresholds, but the tool's default must not be weakened. This is exactly the "rules of the system" leverage point Seren identified — and it's a gatekeeper leverage point, not a feedback leverage point.

4. **The immutable precision threshold addresses Quinn's concern too.** Quinn proposed a dual threshold (per-firing precision AND per-scan FP cap). The per-scan FP cap is a sensible additional constraint, but it should operate as a separate alarm — "your codebase has too many findings for this rule to be useful here" — rather than as a mechanism to demote a precise rule. A rule that is 97% precise but produces 30 findings per scan doesn't have a precision problem; the codebase has a boundary architecture problem. Don't weaken the rule; fix the code.
